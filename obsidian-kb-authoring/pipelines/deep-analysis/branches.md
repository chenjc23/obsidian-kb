# branches — 分支流程逐个展开

先 `scaffold flow --repo {repo} --topic {主题} --member 分支主题` 拿骨架,填好后 Write 到 `分支主题.md`——**此文件必生成**,作本流程分支总览(exists 闸门自动判完成,无需手动 `pipeline done`)。

1. 回到主干每个分支标记;主动从调用树、条件表达式、错误路径、协议/消息分发、状态机迁移、重试、回滚、超时、下游回调发现更多关键分支。
2. 按重要性与风险排序,完整分析每个关键分支,记入 `分支主题.md`。
3. 写在哪按体量决定(避免过度拆文件):简单分支就地写在 `主干流程.md` 对应步骤下、在 `分支主题.md` 引用;够分量的(逻辑链长、深嵌套、高风险、业务关键、被复用)可再提取独立分支页(按实际分支主题命名 Write,一个分支一份),但 `分支主题.md` 始终是必生成的总览。
4. 本流程无够分量的关键分支时,`分支主题.md` 仍生成,标 `confidence: low`,正文一句注明「关键分支均就地写在主干流程 / 本流程无显著分支」。

这是决定「写在哪」不是「要不要分析」——每个关键/高风险分支都必须覆盖。独立成文件的分支需补来源双链(主干 + 调用树节点 + 兄弟/嵌套分支),规则见 `references/link-contract.md`。关键分支未覆盖须以源码证据在 `自查报告.md` 记为缺口,不得用「其他分支类似」敷衍。
